## 一、问题定义与整体架构

### 🔑 1. 恶意代码检测的序列分类建模

**一句话**：把恶意代码检测从"给一个置信分数"重新定义为"序列分类 + 行号定位"的联合任务。

传统方法的形式化很简单：给定一个文件，输出 P(malware)∈[0,1]。但本文在问题定义上多加了三个约束：

- **跨文件上下文感知**：分析对象是整个项目，而非单个文件。单文件分析无法应对"分散潜伏、协同触发"的供应链攻击。
- **语句级定位**：不仅判定是否恶意，还需输出恶意行号范围 L={(lineᵢ, attentionᵢ)}。
- **可解释性**：通过注意力权重为判定提供可追溯的依据。

形式化定义：给定代码块 c 及其项目级上下文 C，模型输出标签 y∈{0,1}，同时输出定位信息 L={(行号, 注意力贡献度)}。

---

### 🔑 2. 五模块整体架构

**一句话**：系统由多语言解析、Graph-RAG 上下文组装、双通道融合分类、注意力定位、对抗训练五个模块串联组成。

五个模块的协作链路：

```
项目目录 → [模块1: Tree-sitter解析+切块] → [模块2: Graph-RAG上下文组装]
         → [模块3: 双通道融合分类] → [模块4: 注意力定位] → 检测报告
                                  ↕
                          [模块5: FreeLB对抗训练 + Focal Loss]
```

每个模块解决一个具体瓶颈：模块1解决跨语言统一解析；模块2解决跨文件上下文断裂；模块3解决语义与结构信息互补；模块4解决可解释定位；模块5解决类别不平衡与鲁棒性。

## 二、代码解析与表示（支撑概念）

### 3. Tree-sitter（通用技术）

**一句话**：GitHub 开源的增量式代码解析框架，生成保留源码全部细节的具体语法树。

三个核心特性：
- **增量解析**：局部修改只更新受影响子树，处理大型仓库不成为瓶颈。
- **容错解析**：遇到语法错误不崩溃，将错误区域标记为 ERROR 节点，其余照常解析。恶意脚本经常故意包含残缺片段，这个特性至关重要。
- **多语言支持**：通过 grammar 定义文件覆盖 40+ 语言。

---

### 4. CST vs AST（通用概念）

**一句话**：CST 保留源码所有细节（逗号、括号、注释都在），AST 只保留能运行的"骨架"。

为什么本文选 CST：
1. 混淆代码的痕迹（异常括号嵌套、奇怪标点分布）在 AST 中会被丢弃
2. CST 节点精确记录行号和列号，为行级定位提供物理依据
3. 注释也被保留——恶意代码的注释模式可能提供线索

> 论文中的比喻：AST 只收答题卡，CST 把草稿纸也保留了。

---

### 5. Magika（通用工具）

**一句话**：Google 2024 年发布的深度学习文件类型识别工具，基于内容而非扩展名判断文件类型。

从文件开头、中间、末尾各取 512 字节，用深度学习模型识别 200+ 种文件类型。准确率 99%+，单次推理约 5ms，模型仅约 1MB。

在本文中的作用：攻击者常把恶意 Python 代码保存为 `.txt` 或 `.md` 绕过扩展名过滤，Magika 能基于内容识别真实语言类型。

## 三、多语言解析与切块（🔑 本文设计）

### 🔑 6. 细粒度切块策略

**一句话**：将源码文件拆分为函数级到语句级的代码块，物理行号从解析开始就挂在每个块上，保证全程可溯源。

切块粒度控制在函数级到语句级之间——太粗（整个文件一个块）则信息密度过高，模型难以精确判断；太细（每个语句一个块）则上下文碎片化，语义信息不完整。

---

### 🔑 7. 语言差异化切块

**一句话**：不同编程语言有不同的语法结构和恶意代码模式，切块策略因语言而异。

| 语言 | 切块入口 | 特殊处理 |
|------|----------|----------|
| **Python** | function_definition + expression_statement | 模块顶层游离代码段（恶意 Python 常直接在模块级别执行 `os.system()` 或 `eval()`） |
| **JavaScript** | function_declaration + arrow_function + expression_statement | 专门检测 IIFE 模式（`(function(){...})()` 是 JS 混淆后门的常见形式） |
| **TypeScript** | 同 JS，扩展 grammar | 作为 JavaScript 超集处理 |
| **Java** | method_declaration + static 初始化块 | 额外关照包含命令执行、文件操作等敏感 API 的方法 |
| **C++** | function_definition | 标记包含系统调用、内存操作等敏感操作的函数 |

设计依据：恶意代码的"藏身之处"因语言习惯而异。Python 的模块级直接执行代码、JavaScript 的 IIFE 后门、Java 的 static 初始化块——这些是各自语言中最容易被利用的入口，切块策略必须覆盖。

---

### 🔑 8. CodeChunk 数据结构

**一句话**：每个代码块在系统内部用一个统一的八元组结构表示，字段覆盖了解析、图构建、特征提取和定位回溯的全流程需求。

```
CodeChunk = {
    chunkId,           // 唯一标识
    filePath,          // 所属文件
    language,          // 编程语言
    text,              // 代码文本
    startLine,         // 起始物理行号
    endLine,           // 终止物理行号
    imports,           // 导入的外部依赖列表
    definedSymbols,    // 本块定义的符号集合（函数名、类名、变量名）
    referencedSymbols, // 本块引用的外部符号集合
    staticFeatures,    // 原始静态特征值（归一化前）
    featureVector      // 归一化后的特征向量
}
```

关键设计考量：
- **startLine/endLine** 从解析阶段就挂载，贯穿整个管线，保证定位回溯不丢失行号信息
- **definedSymbols/referencedSymbols** 在 CST 解析阶段预提取，Graph-RAG 图构建阶段直接复用，避免重复遍历语法树
- **staticFeatures 与 featureVector** 分离存储，因为归一化需要基于全数据集统计量，无法在解析阶段完成

---

### 🔑 9. 行号映射与全程溯源

**一句话**：物理行号从 Tree-sitter 解析开始记录，贯穿切块、组装、编码、分类、定位五个阶段，保证最终定位结果是原始文件的可读行号。

溯源链路：
```
Tree-sitter CST节点 → (startLine, endLine) → CodeChunk → 组装文本中的偏移 → 
Tokenizer offset_mapping → token位置 → 注意力权重 → 逆向映射回原始行号
```

行号不丢失的保障：每个阶段的操作（截断、拼接、tokenization）都同步更新 offset_mapping，确保 token 位置与原始文件行号的对应关系始终可逆推。

## 四、预训练模型与 NLP 技术（支撑概念）

### 10. UniXcoder（通用模型）

**一句话**：Microsoft ACL 2022 提出的代码预训练模型，"一个模型，多种能力"。

基于 RoBERTa 的 12 层 Transformer 编码器，隐藏维度 768，12 个注意力头。核心创新是通过**前缀适配器**和**注意力掩码**在三种模式间切换：编码器模式（代码理解）、解码器模式（代码补全）、编码器-解码器模式（代码生成）。

多模态预训练将 AST 线性化为 token 序列与注释并列输入，同时跑 MLM 和对比学习。

本文选择理由：单塔架构避免双塔的模态对齐偏差；多语言预训练（6-9 种语言）降低多语言部署成本；AST 感知预训练让其天然对代码结构敏感。

---

### 11. CodeBERT / GraphCodeBERT（对比模型）

**一句话**：CodeBERT 是微软首个代码双塔预训练模型（EMNLP 2020），GraphCodeBERT 在其基础上引入数据流图（ICLR 2021）。

- **CodeBERT**：双塔架构分别编码自然语言和代码，MLM + RTD 预训练。问题是双塔天然存在模态分离偏差。
- **GraphCodeBERT**：在 CodeBERT 基础上引入数据流图感知变量间数据依赖。但对动态类型语言（Python、JS）静态抽取 DFG 的精度有限。

---

### 12. [CLS] Token Pooling（通用技术）

**一句话**：取序列第一个 token 的输出向量作为整个输入的语义表示。

[CLS] 在预训练阶段通过"下一句预测"被训练成序列信息的聚合点。本文实验表明在代码分类任务上 [CLS] Pooling 比 Mean Pooling 高约 1.5 个点 F1——因为 Mean Pooling 把每个 token 平均化后，eval 调用、危险 import 这些真正的强信号被大量普通 token 稀释了。

---

### 13. Tokenization 与 offset_mapping（通用技术）

**一句话**：Tokenizer 把代码文本切分成 token 序列，offset_mapping 记录每个 token 对应原始文本的起止字符位置。

UniXcoder 使用 BPE 分词。offset_mapping 是定位算法的关键桥梁——注意力权重在 token 级别，但最终需要输出源码行号，offset_mapping 提供 token→字符→行号的映射路径。

## 五、双通道融合分类（🔑 本文设计）

### 🔑 14. 双通道融合分类架构

**一句话**：语义通道（UniXcoder）理解"这段代码想干什么"，结构通道（37维静态特征）提供"这段代码长什么样"的硬指标，两者互补。

两个通道的信息类型不同：
- **NLP 语义通道**：UniXcoder → [CLS] Pooling → 768 维语义向量。捕获代码的深层语义意图——"它在做网络请求然后执行返回的数据"。
- **静态特征通道**：37 维原始特征 → 自适应归一化 → 投影层(37→128) → 128 维结构化向量。提供确定性安全信号——"它调了 3 次 eval，圈复杂度 18，信息熵 6.2"。

分类决策头：融合向量 → MLP(896→512) → GELU+LN+Dropout → MLP(512→256) → GELU+LN+Dropout → Linear(256→2) → Softmax。

设计动机：语义和结构是两类不同性质的信息。语义看"意图"，结构看"证据"。只靠语义容易被混淆代码欺骗；只靠结构则无法理解执行逻辑。两者从不同视角看同一段代码，信息天然互补。

---

### 🔑 15. 门控融合机制

**一句话**：根据输入样本的特征，动态决定语义通道每个维度的信息应该"放行"多少。

数学过程：
```
g = σ(W·[h_nlp; h_static_proj] + b)    // W∈ℝ⁷⁶⁸ˣ⁸⁹⁶, b∈ℝ⁷⁶⁸
h_gated = g ⊙ h_nlp                      // 逐元素乘法，g∈(0,1)⁷⁶⁸
```

直观理解：拼接后的 896 维向量过一层全连接 + Sigmoid，生成 768 维的门控向量 g，每个分量取值 (0,1)，逐元素控制语义向量的对应维度。对于直白的 `os.system("rm -rf /")`，语义通道自己能判断，门控增强语义权重；对于经过变量重命名和字符串加密的混淆代码，门控自动降低语义贡献，让静态特征（危险函数计数、混淆评分）占更大分量。

消融实验证据：去掉门控直接拼接，F1 下降 2.33 个百分点，精确率下降 5.29 个百分点——说明直接拼接无法有效利用静态特征，门控的价值恰在于输入自适应的动态调节。

## 六、静态特征通道（🔑 本文设计）

### 🔑 16. 37 维静态特征向量

**一句话**：从文本统计、CST 结构、程序依赖图和对抗模式四个维度提取的 37 项量化指标。

| 维度类别 | 维数 | 包含内容 | 设计意图 |
|----------|------|----------|----------|
| 文本统计特征 | 10 | 圈复杂度、嵌套深度、代码行数、危险函数调用频次、字符串香农熵、Base64 模式出现次数、混淆评分、导入语句数量、函数定义数量、注释比例 | 捕获代码的宏观复杂度、危险模式密度和混淆痕迹 |
| CST 结构特征 | 21 | 节点总数、最大深度、调用表达式数、字符串字面量数、标识符数、括号嵌套最大深度、标点符号总数、左右括号数、叶子节点数和比例、平均子节点数、标识符长度总和、独特标识符数和比例、单字符标识符数、字符串总长和平均长度、调用链最大深度 | 从语法树形态检测异常——混淆过的代码 CST 形态往往不自然 |
| 图结构特征 | 3 | 入度、出度、逻辑深度 | 反映代码块在项目调用网络中的位置——被频繁调用的入口节点更值得关注 |
| 对抗特征 | 3 | 动态代码构造计数、反射调用计数、间接调用深度 | 专门针对恶意代码绕过静态检测的常用技法 |

特征选择的核心原则：每项特征都经过可解释性筛选——必须能说清楚"这个数字高意味着什么"，而不是简单堆砌。

### 🔑 17. 混淆评分

**一句话**：将变量名长度分布、括号嵌套异常度和 CFG 连通模式综合为一个标量，判断代码被混淆的概率。

三个子指标：
1. **变量名长度分布**：混淆后的代码单字符变量（`a`, `b`, `c`）和规律性命名的变量（`_0x1a2b`）占比异常高。正常业务代码中单字符变量极少超过 5%。
2. **括号嵌套异常度**：混淆工具常在括号上做文章——JSFuck 风格代码的括号密度远超正常代码。
3. **CFG 连通模式**：正常条件分支形成菱形结构；控制流平坦化后呈现"全连通星型"——所有基本块指向同一个分发块。

综合方式：三个子指标各算一个 Z-score，加权求和后映射到 [0,1]。

---

### 🔑 18. 字符串香农熵在恶意检测中的应用

**一句话**：通过字符分布的随机程度区分正常字符串（英文、JSON、URL）和加密/编码后的 payload。

公式：**H = -Σ(pᵢ · log₂(pᵢ))**

经验阈值：
- 正常代码字符串：熵 4.5-5.5（英文单词、配置项、URL 路径）
- Base64/加密后字符串：熵 > 6（字符接近均匀随机分布）

误区提醒：压缩过的二进制资源（内嵌图标等）熵值也高。需要配合"附近是否存在解密函数调用"来判断——如果高熵字符串旁边有 `base64.b64decode()` 或 `xor()`，那就是强恶意信号。

## 七、Graph-RAG 上下文组装（🔑 本文设计）

### 🔑 19. Graph-RAG 上下文组装

**一句话**：以程序依赖图（PDG）为检索底层结构，通过调用链回溯为每个代码块补全跨文件上下文信息。

**重要区分**：本文的 Graph-RAG = **Program-Dependence-Graph-based Retrieval-Augmented Context Assembly**，与 Microsoft 2024 年提出的 GraphRAG（基于知识图谱的 RAG，用 LLM 从文本中抽取实体和关系构建图）不是同一概念。

本文的"图"是程序依赖图（PDG），节点是代码块，边是调用/依赖关系，基于 Tree-sitter 的符号分析确定性地构建——不依赖 LLM 抽取。

核心步骤：
1. **符号提取**：解析阶段预提取每个代码块的 definedSymbols 和 referencedSymbols
2. **节点构建**：每个代码块 → PDG 节点，属性包括出入度、逻辑深度、符号集合
3. **边构建**：两种边——调用边（A 引用 B 定义的符号）、依赖边（A 导入的模块含 B）
4. **边权重**：w(i,j) = |refs(i) ∩ defs(j)| / |refs(i)|，权重越高调用关系越紧密
5. **触发召回**：检测到危险函数时自动触发 K-Hop 反向回溯

---

### 🔑 20. 三级降级召回策略

**一句话**：优先确定性召回，逐步降级为统计召回，确保在各种场景下都能拿到有效上下文。

| 级别 | 触发条件 | 方法 | 可靠性 |
|------|----------|------|--------|
| **第一级** | 代码块含危险函数 | K-Hop 拓扑召回（沿调用边反向回溯，默认 K=3） | 最高：基于程序语义的确定性关系 |
| **第二级** | 无条件触发或无危险函数 | 精确符号匹配（在全局符号表中查找每个导入符号的定义节点） | 高：基于命名空间匹配 |
| **第三级** | 图构建失败或前两级为空 | TF-IDF 语义召回（向量相似度检索） | 兜底：统计相似度，有上下文断裂风险 |

设计哲学：精确符号匹配和 K-Hop 拓扑召回基于程序语义的确定性关系，召回结果与目标代码有明确的调用依赖——避免传统向量检索"召回了但语义不相关"的上下文断裂。

第一级中的 K 默认取 3 的理由：实验发现 K>3 时召回节点数指数增长但有效信号递减（远距离调用链大多是噪声），K<3 则可能漏掉关键的上游定义节点。

---

### 🔑 21. 长文本组装格式

**一句话**：用特殊标记将目标代码和召回的上游上下文组装为统一序列，作为模型输入。

组装格式：
```
[LANG:python] <CONTEXT> [FILE:utils.py] def decrypt(p): return base64.b64decode(p) </CONTEXT>
[SEP] <TARGET> [FILE:main.py] import utils; exec(utils.decrypt(payload)) </TARGET>
```

标记含义：
- **[LANG:xxx]**：编程语言，让模型感知语言上下文
- **[PATH:xxx]**：调用路径（CALL=直接调用，IMPORT=导入依赖）
- **\<CONTEXT\>...\</CONTEXT\>**：Graph-RAG 召回的上游代码
- **\<TARGET\>...\</TARGET\>**：待检测的目标代码块
- **[SEP]**：上下文与目标之间的分隔符

截断策略：总长度超 1024 token 时，优先保留与目标代码直接相关的文件；远程间接依赖优先截断。这是"语义就近原则"——直接调用者比间接调用者携带更多判断所需信息。

---

### 🔑 22. 重叠滑动窗口推理

**一句话**：超长代码超过模型最大输入长度时，切成多个重叠窗口独立推理，用 Hanning 窗加权平均得到最终预测。

参数：窗口 1024 token，步长 256 token，重叠 768 token（75% 重叠率）。

加权公式：**FinalLogits = Σ(ωᵢ · Logitsᵢ) / Σωᵢ**

Hanning 窗（余弦型加权函数）的作用：窗口中心区域权重最大、边缘权重最小。边缘处的预测本身不可靠（代码被切断，上下文不完整），Hanning 窗让这些低置信度预测在最终融合中贡献较小。

## 八、注意力定位（🔑 本文设计）

### 23. Attention Rollout（通用技术 + 🔑 本文改进）

**一句话**：递归累积 Transformer 所有层的注意力矩阵，得到从输入到输出的完整注意力传播路径。

由 Abnar & Zuidema (ACL 2020) 提出。

核心公式：
```
Ãₗ = 0.5·Aₗ + 0.5·I      // 每层注意力 + 残差连接（0.5:0.5 是 Rollout 的经验比例）
Rollout = Ã₁ × Ã₂ × ... × Ã₁₂  // 12 层逐层累积
```

为什么要做 Rollout：直接看最后一层的 [CLS] 注意力只能看到"终态"——信息经过 12 层混合后最终定格在什么位置。但中间层的信息传播路径被忽略了。第一层可能关注"附近的词"，第五层关注"语义相关但距离较远的调用"，第十二层关注"对分类决策最关键的位置"。Rollout 把整条传播路径展开，归因更完整。

本文实验结果：Rollout 把 Top-3 命中率从直接 [CLS] 注意力的 75.00% 提升至 84.21%。

---

### 🔑 24. CST 语义边界感知平滑

**一句话**：用 Tree-sitter CST 的语法边界信息约束注意力平滑——平滑操作只允许在同一语法结构内部进行，防止权重跨函数、跨类扩散。

**这是本文定位算法的核心创新点**。

问题根源：原始 Attention Rollout 不关心代码的语法结构。恶意函数 A 和相邻的无关函数 B 在 token 序列中是连续的，注意力权重可能在两者之间自由流动——A 的高注意力扩散到 B 上，导致定位偏移。

本文方案的三步操作：
1. **边界提取**：利用 Tree-sitter CST 在解析阶段记录的节点类型（function_definition、class_declaration 等）和起止位置，识别语法边界。
2. **边界约束**：滑动窗口平滑时，窗口范围被 CST 边界裁剪——不越过当前语法结构的起止位置。
3. **权重重分配**：被边界截断的窗口，其内部权重重新归一化，确保平滑后的分布总和不变。

直观效果：如果注意力集中在恶意函数内部的某处，平滑后权重仍在同一函数内部，不会"泄漏"到相邻函数。

实验证据：CST 平滑将 Top-3 命中率从 Rollout-only 的 84.21% 提升到 89.47%，行级精确率从 66.92% 提升到 72.36%。精确率的提升（+5.44pp）大于召回率的提升——说明 CST 平滑的主要价值是减少"定位到了但位置不精确"的情况。

---

### 🔑 25. 逆向行号映射（六步流程）

**一句话**：将 token 级别的注意力权重转换为原始源码文件中的物理行号。

完整的六步逆向映射：

```
Step 1: 剔除特殊token
  └── [CLS]、[SEP]、[PAD]、[LANG]、[PATH]、<CONTEXT>、</CONTEXT>、<TARGET>、</TARGET>
      这些 token 不对应实际代码，注意力权重直接置零

Step 2: 滑动窗口平滑
  └── 将相邻 token 的注意力分数合并（针对 sub-word token，如 "cmd"→"c"+"##md"）
      ŝᵢ = (1/|W|) · Σ_{j∈W(i)} sⱼ

Step 3: 选取 Top-K 高分 token
  └── K 默认取 5，作为候选恶意位置

Step 4: 字符坐标映射
  └── 通过 offset_mapping 查询每个 token 在组装文本中的 (start, end) 字符位置

Step 5: 文件归属判断
  └── 根据组装时记录的元数据（TARGET 区域偏移、CONTEXT 区域偏移）
      判断高分字符落在哪个文件

Step 6: 字符偏移 → 行号
  └── 在所属文件中做二分查找（每行起始字符偏移量预先索引）
      输出: (文件路径, 行号, 代码片段, 注意力贡献度)
```

## 九、模型训练策略（🔑 本文组合应用）

### 26. Focal Loss（通用技术 + 🔑 本文针对性配置）

**一句话**：专门解决类别不平衡的损失函数，降低大量易分类样本的损失权重，让模型聚焦于难分类的少数类。

由 Lin et al. (ICCV 2017) 提出。公式：**FL(pₜ) = -αₜ(1-pₜ)ᵞ·log(pₜ)**

本文配置：α=0.25, γ=2

为什么这样设：
- **α=0.25**（而非"恶意占比 13.11% 对应的 0.13"）：因为恶意样本虽是少数但更重要（漏报代价 >> 误报代价），故意给了一个高于其自然比例的权重。
- **γ=2**：最常用的取值。γ=1 降权不够，γ=5 则过度压制易分类样本导致模型不稳定。

消融证据：去掉 Focal Loss 后，召回率从 99.02% 骤降至 54.55%，FPR 从 0.11% 升至 5.07%。

---

### 27. FreeLB 对抗训练（通用技术 + 🔑 本文应用）

**一句话**：在词嵌入层注入对抗扰动，通过 K 步梯度累积增强模型对混淆代码的鲁棒性。

由 Zhu et al. (ICLR 2020) 提出。

核心流程：
1. 初始化扰动 δ=0（与词嵌入同维度 n×d）
2. K 步对抗上升：每步沿梯度方向更新 δ→损失增大→投影回 ε-球
3. **关键设计**：K 步全部累加模型参数梯度，只在最后做一次更新

与标准 PGD 的核心区别：PGD 丢弃前 K-1 步的参数梯度，FreeLB 全部利用——等效于 batch size 扩大 K 倍，同时每一步扰动不同增强了对抗多样性。

另一关键细节：K 步中 Dropout 掩码固定不变——避免不同网络结构引入噪声梯度。

在本文中的价值：FreeLB 生成的对抗样本模拟了代码混淆（变量重命名、字符串加密）对嵌入层的扰动效果，使模型在训练中就见过"被干扰过的代码"，提升对真实混淆攻击的鲁棒性。

---

### 28. LoRA 低秩适应（通用技术 + 🔑 本文配置）

**一句话**：冻结原模型参数，只训练少量新增的低秩矩阵，显著降低显存占用。

由 Hu et al. (ICLR 2022) 提出。核心思想：ΔW = BA，B∈ℝᵈˣʳ，A∈ℝʳˣᵏ，r≪min(d,k)。

本文配置：r=16, α=32，作用于 UniXcoder 的 query、key、value、dense 四个模块。

选择这四个模块的理由：它们是自注意力机制中信息交互最密集的部分。query/key 决定"关注谁"，value 决定"提取什么信息"，dense 负责输出融合。仅在这四个模块注入 LoRA 即可覆盖注意力计算的核心路径，无需修改 FFN 层。

---

### 29. Warmup + 早停（通用技术 + 🔑 本文配置）

**Warmup**：训练初期用较小学习率逐步增大到预设值，避免初期参数剧烈振荡。本文 warmup_ratio=0.1，学习率从 0 线性增加到 2e-5。

**早停**：监控验证 F1，连续 patience=3 个 epoch 未改善（提升 < threshold=0.001）则停止，保存最佳模型。选择 F1 而非 Loss 作为监控指标，因为 F1 直接反映最终业务目标。

## 十、数据集与评估（🔑 本文设计）

### 🔑 30. 项目组划分策略

**一句话**：按项目根目录划分训练/验证/测试集，确保同一项目的所有代码块只出现在一个集合中。

为什么不能随机按代码块划分：同一项目内的代码块高度相关（共享同一套变量命名风格、同一套代码规范）。如果代码块 A 在训练集，同一项目的代码块 B 在测试集，模型可能通过"记住了项目风格"而非"学到了恶意模式"来做对——这是数据泄露。

本文解决方案：以项目根目录为划分单元。恶意 92 个项目按 64:14:14 划分，良性 137 个项目按 96:21:20 划分（约 70:15:15）。

但论文也诚实指出：不同项目之间的代码风格差异、功能领域重叠等因素可能引入更隐式的分布偏移，需要在更大规模数据集上验证。

---

### 🔑 31. 变量重命名数据增强 + 危险函数保护

**一句话**：将代码中的标识符替换为等价但形式不同的名称来扩充恶意样本，同时对 eval、exec 等危险函数名做豁免保护。

核心矛盾：数据增强是为了增加样本多样性，但如果不加保护地把变量名乱改——把 `eval` 重命名为 `_x1234`——模型就学不到"eval 是危险的"这个关键信号了。

保护机制：在重命名前检查标识符是否在危险函数白名单中：
- Python：eval, exec, compile, __import__, os.system, subprocess, pickle.loads, marshal.loads, ...
- JavaScript：eval, Function, child_process.exec, Buffer.from, atob, ...
- Java：Runtime.exec, ProcessBuilder, Class.forName, ScriptEngine.eval, ...
- C++：system, popen, dlopen, ...

增强效果：恶意代码块从 3762 增至 15048（4 倍），占比从 3.63% 提升至 13.11%。

---

### 🔑 32. MD5 + SimHash 两级去重

**一句话**：先用 MD5 精确去重，再用 SimHash 语义近似去重；样本量超 5 万时自动降级为纯 MD5。

- **MD5 去重**：完全相同 hash 的代码块直接删除。速度快但只能抓完全重复。
- **SimHash 去重**：计算 SimHash 指纹，汉明距离 ≥ 阈值时判为不相似。能抓近重复（如仅改了变量名的两个版本）。

自动降级逻辑：SimHash 的两两比对复杂度 O(n²)，超过 5 万样本时计算量过大。此时只做 MD5 去重。

## 十一、评估指标（支撑概念）

### 33. 七项评估指标

| 指标 | 公式 | 关注什么 |
|------|------|----------|
| Accuracy | (TP+TN)/All | 整体判别能力（但不平衡数据集中参考价值有限） |
| Precision | TP/(TP+FP) | 误报控制——判为恶意中有多少是真的 |
| Recall | TP/(TP+FN) | 漏报控制——真的恶意中有多少被检出 |
| F1 | 2PR/(P+R) | 精确率与召回率的调和平均 |
| MCC | (TP×TN-FP×FN)/√[...] | 唯一同时考虑四类结果，对不平衡鲁棒 |
| ROC-AUC | — | 不同阈值下的综合区分能力 |
| FPR | FP/(FP+TN) | 良性被误判的比例——安全场景上线关键指标 |

为什么不用 Accuracy 做主要指标：全预测"良性"就能拿 86.89% Accuracy，但漏掉所有恶意样本——毫无价值。

## 十二、静态程序分析（支撑概念）

### 34. 圈复杂度、嵌套深度、控制流平坦化、IIFE

已在上一版完整覆盖，此处保留条目索引：

- **圈复杂度**：M = E−N+2P，简化 CC = 1 + Σ(每个决策点出现次数)。决策点定义因语言而异。
- **嵌套深度**：控制结构最大嵌套层数，正常建议 ≤3-4 层。
- **控制流平坦化**：将 if/else 替换为大 switch-case + 状态变量，CFG 呈现"全连通星型"。
- **IIFE**：`(function(){...})()`，JS 混淆后门的常见载体。
